>     2026-05-23 凌晨：steam-cli 成为核心数据来源。
>     02:41 Integrate steam-cli for Vercel deployment，03:03 Organize assets and load public preview dynamically。这很关键，项目不再只是前端/手写数据，而是开始把 steam-cli vendored 进来，服务 Vercel 部署和运行时数据。

>     旧 docs/current-status.md 写得很直白：产品方向已经从 “wishlist releases only” 移到 **Steam to Calendar**。同一天 Git 里有 Build Steam sale calendar experience、Redesign Steam sale calendar、移动端 UI、list view、event range 等提交。GStack learning 也记录了你偏好的定位：直接说“把 Steam 游戏和事件放进系统日历”，不要抽象品牌词，也不要只叫 Sale，免得窄化。

>     这一天有 35 个 commit，是整个项目最密集的一天。包括 Refine Steam to Calendar UI positioning、setup panel、ready step、subscribe CTA、mobile spacing、region/currency、game image ratio、Vercel analytics 等。这个阶段产品从“能用”变成“像一个真的应用”。
>
>     2026-05-28 凌晨：Claude 记录里开始以 steam-to-calendar 作为新项目路径做代码质量审查。

>     2026-05-29 22:36：正式重建为 Steam to Calendar。
>     f4ecc8c Rebuild as Steam to Calendar 是大节点：改产品名、删除旧 docs/design assets、移除 src/lib catch-all，把架构拆成 app / features / domain / integrations / server / shared。品牌资源也从 wishlist-in-calendar-logo.png rename 成 steam-to-calendar-logo.png。
>
>     2026-05-31 凌晨：再次打磨公开定位。
>     GStack 新项目 nickxudotme-steam-to-calendar 里生成了 Deal-first positioning for Steam to Calendar。核心用户原话是：这个产品“能无感在日历查看最近关注的游戏打折情况”。这时定位变成：wishlist 是输入方式，不是产品本身；核心价值是低噪音地把关注游戏的折扣/发售/预购/活动放进系统日历。

>     2026-06-01：wishlist preview 开始流式化。
>     17efff5 Stream wishlist preview updates 增加 NDJSON streaming，让连接 wishlist 后可以先逐步显示 wishlist games，再等最终 calendar preview。这是体验从“等接口返回”变成“逐步有反馈”的节点。
>
>     2026-06-02 到 2026-06-03：移动端、地区、币种、文案细节收口。
>     这里包括移动端 intro sheet、locale selector、wishlist rate-limit 友好文案、历史价格币种格式化、币种 mismatch note、region refresh 时显示 loading currency。.gstack/browse-startup-error.log 也在 2026-06-03 00:02 留下一条浏览器启动失败记录，和当时本地/浏览器验证不太顺的记忆对得上。
>
>     2026-06-03 到 2026-06-04：从产品变成可观测的生产服务。
>     先加 observability events，再切到 Umami，后来改成 self-hosted Umami，加 server analytics、ops reporting script、session replays、preview telemetry。到 2026-06-04，项目已经不只是能生成日历，而是能看部署和用户行为信号了。
